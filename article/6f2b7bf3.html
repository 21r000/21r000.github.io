# 完整分析法

PEID 查看基本信息

# 没加壳

入口点： 00001130

文件偏移： 00001130

EP 段： .text

首字节： 55,8B,EC,6A

# 加壳后

入口点： 00013000

文件偏移： 00003E00

EP 段： .pediy

.pediy 外壳程序，入口点变为新区段的 v.offset （偏移）上去执行.pediy 的代码，执行完之后跳转到 .text 区段执行原来的入口。

首字节： 60,E8,C2,00

# OD 分析

将加壳后的 .exe文件 用 OD 打开

执行程序后停在 CALL RebPE.004130C8 位置，断点记录

上面的 pushad 起保存环境的作用，这个 call 起取地址的作用

接下来重新调试后 f9 跳转，然后 f7 跟进就是我们要分析的主模块部分了

上面一部分就是壳的入口点

我们 f8 一步步往下走，在如下位置可以看到一个注释为 kernek32.getprocaddress 的获取函数地址的 api

在右下角栈中可以看到参数的值

此时 getprocaddress 返回一个值放到 eax 中，而在下一行中通过 mov 将 eax 的值给 ebp + b8

跟进到 ebp + b8 部分，此时可以看到 call 的 ebp + b8 的位置存储的就是 kernel32.virtualalloc 的函数地址，而 ebp + b8 通过上面四个 push 可以看出它有四个参数

继续往下走可以发现一个 jmp，而这个 jmp 则起到转到壳的第二个部分继续执行的作用

f8 继续执行则会进行跳转，此时跳转到的位置就是壳的第二部分的位置

如下 SUB EDX,5 获取外壳第二部分的基址

执行完可以发现 edx 的值发生变化，也即壳的第二部分的基地址

继续往下有一个 je 短跳，用于处理 dll动态链接库 文件

继续往下走到如图所示位置进行断点

进行 f8 执行到 CALL DWORD PTR SS:[EBP+34D] 可以看到注释为 KERNEL32.VirtualAlloc ，而 VirtualAlloc 的作用则是申请内存，进行读写

获取解压区块的 rva ，解压完就可以看到 edi 中的值显示出来

继续往下，开始解压

跟进查看

继续往下执行，获取原来区块大小，也即解压数据的大小

继续往下即可看到 ecx 的大小，也即原来数据的大小

到此，程序已经获取到数据的大小，起始位置，rva，申请内存读写等条件

接下来程序开始写回去，往下 rep 重复操作，将解压后的数据写入内存

继续往下存在一个 call ，注释为 KERNEL32.VirtualFree ，释放内存，防止内存泄露

如此解压部分完成，而在壳的加载流程中还存在一个 iat 的填充

在 000300A9 位置存在一个 je 跳转，通过上面 cmp 的对比结果判断是否跳转，据此判断这个位置为 iat 填充

修改 je 指令为 jmp 指令进行跳转

进入 iat 填充部分

or 判断 eax 是否为 0，如果值为 0 则表示导入表没有加密，否则进行过加密

jnz 跳转

进入对转储之后的导入表初始化部分

通过 jnz 来回跳转对 iat 进行填充

iat 填充完成后开始修复重定位

反 nop ，跳转到 00030233

判断操作系统版本

秉持着在一段结束之后跳转到下一段的原则，找到 jmp

进行跳转，在这个位置会将原来入口点的 rva 取出来

断点执行，可以看到 eax 的值发生变化

恢复环境

PUSH 401130 ，oep 由上面动态生成，将其吐出来之后，将它压入栈中，它即为栈顶

在 return 的时候，栈顶的数据会被弹到 eip 中，再通过返回值就会 jmp 到 eip 的位置，所以将其压入栈中之后 return 到上面 push 的位置上

执行 retn ，进入程序的真正的主程序中

ctrl + a 重新分析，进入真正的入口点，也即 oep 中

# 堆栈平衡法

f8 将 8 个通用寄存器压入栈中，可以看到 esp 的值变为 0019FF54

而有压栈就有弹栈，也即有 push 就有 pop

也就是说在壳处理完所有数据之后就一定会将现场还原

而这个时候 esp 就还会被访问到，此时右键，将 esp 在数据窗口中跟随

然后在第一个数据中字节断点

然后 f9 ，可以看到 PUSH 401130

然后执行分析，进入主函数
